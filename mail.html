Project Proposal

We will be creating a sensor or set of sensors which can detect the presence (or lack of) of mail in an Olin student’s mailbox. This self-contained device will be able to automatically monitor the mailbox’s environment and relay this information to a remote location. This location will receive data from multiple mailboxes, store and interpret this data, and notify mailbox owners appropriately.
As an avenue for future work, the remote location could have an interface to allow mailbox owners to check the status of their mailbox or an administrator to check the status of all mailboxes through a web app or mobile application.

Initial Design Goal — Sensing Device

The sensing device will be entirely self-contained. It can be placed entirely into a mailbox, with no external sources of power or network connectivity. As such, it must operate in such a way to manage battery power efficiently, sending an alert before it runs out of power. Network connectivity should be done in such a way to allow multiple sensing devices to communicate to the same remote location.

Initial Design Goal — Remote Location

The remote location will receive mailbox data from one or more sensing devices. This will parse and store mailbox sensor data. This will operate without a physical connection to the sensor device.

Minimum Deliverable

We will be able to identify the presence (or non-presence) of mail in an Oliner’s mailbox. We will then send an email or text message to notify the owner of the mailbox.

Initial Design

Notification

Our main goal is to notify a user when they receive mail in their mailbox, so our first design decision centered around how to send this notification. We considered text messages, emails, and even Twitter, all of which would be possible using Python. We have chosen email as our first and primary method of user notification. Email is a means of conveying similar information — information about meetings, about orders, and about deliveries — and is the natural manner of letting users know that they have received mail, especially at Olin.

Wireless Connectivity

Our next design decision involved determining how to connect the sensors in the mailbox to our online notification center. A Raspberry Pi, (RPi) which lacks an analog to digital converter (ADC), would make it more difficult to interface with some very useful sensors. Additionally, an RPi running as a mobile sensing unit would consume relatively great amounts of power and lacks a native wireless connectivity option — not ideal for a compact, self-contained sensing unit. A microcontroller, such as the Arduino, is closer to what we need — Arduinos cost less than an RPi and make it easier to interface with sensors. However, wireless connectivity is still a relatively expensive add-on.
Due to these design considerations, we have chosen a Spark Core as the most effective hardware for our purposes. A Spark Core is a tiny microcontroller with tightly integrated wireless connectivity. It allows us to remotely read sensors and run functions using an API.

Server

We determined that we have a need for a server which runs continuously in order to routinely check the mailboxes and send notifications. In future iterations, it is possible that this server need will be met using a Raspberry Pi. However now, in our initial phases of development, using a computer works just fine.

Model

We chose to build a model of one of the Olin mailboxes so we could run tests outside of the mailroom. We used the dimensions from the mailboxes and built a replica out of MDF. By having a 3D model that is mobile, we are able to make sure all of our components fit inside of the mailbox and we can rapidly prototype without having to constantly go into the mailroom, which is especially important if we are working when the mailroom is closed.

Sensors

We are still working on determining the best type of sensor for the job. So far, we have considered pressure sensors and light beam IR sensors. Pressure sensors might not be sensitive enough to account for lighter pieces of mail. IR sensors might be too easily tripped. We plan to experiment with different types of sensors in order to find the best available technology.

Objected Oriented Code

Our class structure was designed to allow for ease of scaling. First, we created a Mailbox object. Each Mailbox object in our current iteration has three attributes: got_mail, owner, and last_notified.

  • got_mail returns a Boolean depending on whether or not there is mail contained in the mailbox.
  • owner contains the contact information of the mailbox owner, in the form of their email address.
  • last_notified returns the time that the user was last pinged. The goal of this attribute is to prevent email spamming of the user. This allows us to only notify the user upon receipt of new mail. This attribute is reset when the user removes their mail.

    • Each Mailbox also has a method called notify_owner() which sends an email when the sensor detects mail in the mailbox of the user. It also records the time that this notification is sent, to be used by the attribute last_notified. Once we expand to have multiple Mailboxes, it will be useful for scaling to be able to access a collection of Mailboxes. In order to do this, we have created a Mailroom class. The Mailroom class currently contains the operating hours of the Mailroom and a collection of Mailboxes.

Final Design

Final Refinement of Our Design

Hardware Additions

In order to mount our sensor inside a mailbox, we 3D printed a stand. We had to make sure that the stand would fit unobtrusively in the mailbox and properly support our sensor. We printed multiple iterations and experimented with positioning the sensor at different angles. Our final iteration involved printing a slot at a 45° angle which accommodated the bend in the sensor. It also allowed the sensor to remain at the correct height in order for mail to activate it. Lastly, we secured the sensor and the mount inside of our mailbox at a reasonable depth that allowed mail to be placed inside and keep it flexed.

Software Additions

Since our last design review, we have eliminated the problem of sending double messages. This was caused by two methods being declared as attributes, without their parentheses. We have also been working on getting our code to read and write .csv files. These files are going to keep track of the state and owner of each mailboxes. Currently, the .csv file that we are writing has two columns, one with the owner’s name and the second with a Boolean that represents whether or not that mailbox has mail. There will be empty rows in our .csv for those students who chose not to participate in our program. By keeping the row number of the .csv consistent with the mailbox number, we are increasing the processing speed of our code. Instead of parsing through many names or numbers, the location of each mailbox is specified by its row number.

Analysis of Outcome

Minimum Deliverable

Our minimum deliverable was defined as being able to identify the presence (or non-presence) of mail in an Oliner’s mailbox and then send an email to notify the owner of the mailbox. We achieved our minimum deliverable within the first week of this six week long project. As it turns out, sending email with Python is a lot simpler than we thought.
What really slowed us down was not having the proper sensor. Due to lack of monitoring of the ordering system put in place at Olin and transition between responsible parties, we were not able to get the sensor that we had hoped to get, a pressure sensitive force sensor. At the beginning of our project, we worked with a light sensor and covered it up to sense the presence of mail. After bringing our sensor problems to our professor, he provided us with three types of sensors to choose from. We thought through design considerations, tested the sensors, and chose to use the force sensitive bendable resistor (flex sensor) that we used in our final project.

Maximum Deliverable

Our maximum deliverable involved creating a web app for users to log into. This proved to have a high level of difficulty, which is why we made it our reach goal. The difficulties occurred due to databasing and the secure saving of passwords.

Design Reflection

Our best decision was choosing to control our system with the Spark Core. We had initially planned to work with a Raspberry Pi, but this proved to be difficult and expensive, due to the lack of availability of the WiFi shield at Olin. We also had some difficulty with selecting a sensor. Looking back, using a beam breaker or a pressure sensor might have been a better option, however we ran into issues when we attempted to order a pressure sensor, which forced us to have to improvise. Our thanks go to Amon for letting us borrow some sensors to test, including the flex sensor that we ended up using for this project.

Division of Labor Reflection

We worked on this project as a group; relatively little code was written without immediate input from other team members. Within the group, each team member was more involved in his or her area of interest. Meg spent much of her time with the Python code that checked mailbox statuses and sent emails. Anders wrote code which created and updated an internet-based data table of every mailbox’s status; he also 3D printed the sensor mount. Ryan was the lead on the Spark Core, the Python class structure, and the creation of a Flask web app. Within meetings we programmed in pairs, switching off between who was typing and who was asking questions or giving advice. Our code is hosted on GitHub for version control.

Bug Report

Our biggest bug was sending multiple emails and spamming users. We thought that we had fixed this after our first design review, but during our second design review, it reared its ugly head yet again. We found it during our design review code presentation. It ended up being a simple problem caused by methods being declared as attributes and lacking the parentheses that define them as methods. It took some searching and reviewing of the code to find the bug, but once we found the origin, it was resolved in a matter of seconds. We could have found it before our design review if we had tested our code before running it in front of our professor and classmates.
